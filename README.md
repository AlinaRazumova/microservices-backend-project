# Microservices Backend Project

Cloud-based task management system designed in a microservices architecture with AWS deployment in mind.

## Project Status

This project is currently in the initial implementation phase.  
The repository contains the base structure of the system, initial API skeletons for the main services, Docker configuration, and project documentation.

## Project Goal

The goal of the project is to design and implement a cloud-based task management system for individual users and small teams.  
The system is intended to support task management, user authentication, role-based access control, and basic CRUD operations through REST API.

## Main Features (MVP)

Planned MVP functionality:

- user registration
- user login and JWT-based authentication
- access control for protected endpoints based on user roles
- creating tasks
- viewing task list and task details
- editing tasks
- deleting tasks
- assigning tasks to users
- managing task status, priority, and deadline
- filtering tasks

## Planned Extensions

Optional future extensions:

- audit log / history of changes
- notifications
- monitoring and centralized logging
- simple web client
- AWS deployment with CI/CD pipeline

## Architecture

The system is based on a microservices architecture.

### Main services

#### Auth Service
Responsible for:

- user registration
- user login
- JWT token generation and validation
- user data access
- role management

#### Task Service
Responsible for:

- creating tasks
- listing tasks
- viewing task details
- updating tasks
- deleting tasks
- assigning tasks to users
- filtering tasks by selected parameters

### Optional service

#### Notification Service
Planned as a future extension of the project.  
It may be responsible for notifying users about task creation, task status changes, and upcoming deadlines.

## API Overview

### Auth Service
Current endpoints include:

- `GET /`
- `GET /health`
- `POST /auth/register`
- `POST /auth/login`
- `GET /auth/me`
- `GET /users`
- `GET /users/{id}`
- `PUT /users/{id}/role`

### Task Service
Current endpoints include:

- `GET /`
- `GET /health`
- `GET /tasks`
- `GET /tasks/{id}`
- `POST /tasks`
- `PUT /tasks/{id}`
- `DELETE /tasks/{id}`
- `PUT /tasks/{id}/assign`

## Error Handling

The project assumes a consistent error response format for API endpoints, for example:

```json
{
  "code": "ERROR_CODE",
  "message": "Error description",
  "details": "Additional optional information"
}
